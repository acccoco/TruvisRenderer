# 代码风格、注释与文档规则

## 1) 代码风格

- 优先使用不可变数据和显式依赖传递，谨慎引入全局可变状态。
- 明确所有权、生命周期和资源边界，避免不必要的共享所有权。
- 错误处理优先健壮性与显式检查，避免把异常当作常规流程。
- 日志聚焦当前函数行为；关键路径保留足够定位问题的信息。
- Rust 文件的 `use` 按来源分组并保持固定顺序：标准库 `std` 在前，其次是第三方依赖，再次是当前项目中的其他 crate，最后是当前 crate 内部路径。不同分组之间保留一个空行。
- 同一 `struct` 的方法应按职责拆分到多个 `impl` 块中，例如创建与销毁、状态更新、渲染提交、资源访问等，避免把不同生命周期阶段或职责混在一个过长的 `impl` 中。
- 单个源文件原则上只承载一个主要 `struct`；其他辅助 `struct` 必须与该主要 `struct` 的状态、资源或流程强相关，避免在同一文件中堆放弱相关类型。
- 避免新增无归属的零散函数；相关行为应收敛到承担对应状态、资源或流程职责的 `struct`/`impl` 中。确需拆分复用逻辑时，应放入已有或明确职责的 helper 模块，而不是散落为 free function。
- Rust 代码使用 `rustfmt`，C/C++ 代码遵循 `clang-format`。
- 不过度抽象，避免 Java 式层层封装；组合优于继承。

## 2) 注释与文档

- 注释解释设计意图、关键实现点、调用契约和全局上下文，不重复显而易见的代码行为。
- 每个 `struct` 都应提供详细注释，说明它承担的职责、核心抽象、关键状态和工作原理；对外使用时需要遵守的调用顺序、生命周期、线程安全、资源释放或同步约束，也应在注释中明确说明。
- 注释使用中文，除非是特定领域术语或代码片段需要英文。
- 涉及生命周期、依赖、同步、线程安全、资源释放和接口契约时，必须说明关键不变量。
- 模块级文档应说明职责边界、非职责、主要抽象、对外契约、重要不变量和依赖关系。
