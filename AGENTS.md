# AGENTS.md

本文件是项目唯一的 AI 协作规则入口，只约束“如何改”。项目结构、分层、时序和模块关系请先看 `docs/ARCHITECTURE.md`；当前实现事实以 `docs/summaries/` 为准。

## 1) 工作方式

- 新增组件、模块或能力前，先查找已有类似实现，优先复用和扩展，避免重复造轮子。
- 改动代码时从全局职责和模块边界出发，避免把相关逻辑拆散到零散 helper 或重复实现中。
- 改动代码后同步维护相关文档；过期设计应提炼到活跃文档或删除，不建立归档目录。
- 不要主动生成单元测试，除非用户明确要求。

## 1.1) 架构预检

涉及新增能力、重构、跨模块调用、资源生命周期、RenderGraph、FFI、shader 或公共接口时，改代码前必须先输出简短架构预检并确认：

- 本次改动属于哪个模块、struct、impl 或明确职责 helper 模块。
- 是否已有相似实现、工具函数、trait、资源封装或文档约定可复用。
- 新增或改变的依赖方向是否符合 `docs/ARCHITECTURE.md` 和 `docs/summaries/`。
- 状态、资源、生命周期和销毁责任由谁拥有。
- 是否改变线程边界、同步不变量、RenderGraph pass 顺序或 GPU resource 契约。
- 需要同步更新哪个 `docs/summaries/` 或模块 README。

如果上述问题无法回答清楚，应先阅读代码和文档，不要直接实现。

## 2) 代码风格

- 优先使用不可变数据和显式依赖传递，谨慎引入全局可变状态。
- 明确所有权、生命周期和资源边界，避免不必要的共享所有权。
- 错误处理优先健壮性与显式检查，避免把异常当作常规流程。
- 日志聚焦当前函数行为；关键路径保留足够定位问题的信息。
- 避免新增无归属的零散函数；相关行为应收敛到承担对应状态、资源或流程职责的 struct/impl 中，或放入已有/明确职责的 helper 模块，保持调用边界集中、可复用。
- Rust 代码使用 `rustfmt`，C/C++ 代码遵循 `clang-format`。
- 不过度抽象，避免 Java 式层层封装；组合优于继承。

## 3) 注释与文档

- 注释解释设计意图、关键实现点、调用契约和全局上下文，不重复显而易见的代码行为。
- 注释使用中文，除非是特定领域术语或代码片段需要英文。
- 涉及生命周期、依赖、同步、线程安全、资源释放和接口契约时，必须说明关键不变量。
- 模块级文档应说明职责边界、非职责、主要抽象、对外契约、重要不变量和依赖关系。

## 4) 渲染项目约束

- 运行渲染示例前参考 `justfile` 中的项目命令。
- 保持现有坐标系约定和渲染管线边界，不随意修改基础约定。
- 触及 shader、C++ FFI 或 RenderGraph 时，优先复用现有模式，不发明新的并行机制。

## 5) 文档职责边界

- `README.md`：面向 GitHub 用户，介绍亮点与上手方式。
- `docs/ARCHITECTURE.md`：当前架构入口、阅读顺序和最高优先级约束。
- `docs/summaries/`：记录当前实现事实，包括分层、生命周期、状态所有权、数据流、线程和资源契约。
- 模块内 `README.md`：说明模块职责、依赖与常见入口。
- `docs/brain-storm/`：记录仍有参考价值的设计讨论与方案评估，不保留归档机制。

## 6) 提交前架构自检

- 新增函数是否有明确归属，避免零散 free helper。
- 是否复用了已有模块边界，而不是重复实现通用逻辑。
- public API、依赖方向、状态 owner、生命周期契约是否发生变化。
- 关键所有权、同步、线程安全、资源释放、FFI、RenderGraph 或 shader 契约是否有中文注释。
- 相关活跃文档是否已同步更新；过期设计应更新或删除。
