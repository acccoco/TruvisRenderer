AssetHub 模块架构分析

AssetHub 是 truvis-asset 的统一入口，当前负责纹理请求去重、异步 IO/解码、GPU 上传推进、 fallback 纹理与状态管理。当前实现已经跨入 GPU 资源和 bindless descriptor 注册边界，这是后续最值得收敛的架构点。

模块定位

truvis-asset 位于架构 L3，职责是资产加载；AssetHub 是该模块对外的门面。 它接收路径加载请求，把磁盘图片推进到 GPU 可采样纹理，并在资源未就绪时提供 fallback。

当前 AssetHub 不是纯 CPU 资产表，因为 new/update/destroy 都会接触 GfxResourceManager 和 BindlessManager。

对外接口

接口 职责 边界特征
AssetHub::new 创建 fallback 纹理、启动 loader、创建 upload manager。 同步注册 fallback SRV，依赖 render-side manager。
load_texture(path) 非阻塞加载入口；路径去重，返回 AssetTextureHandle。 调用后只保证请求已入队，不保证资源 ready。
get_status(handle) 查询 Loading / Uploading / Ready / Failed。 无效 handle 返回 Failed。
get_texture(handle) 返回真实纹理或 fallback。 调用方不需要阻塞等待资源完成。
get_texture_by_path(path) 从路径缓存查 handle，再返回纹理。 路径不存在会 unwrap panic。
update 每帧消费 IO 结果、提交上传、检查 timeline、注册 view/SRV。 核心推进点，运行在 render thread。
destroy 关闭上传管理器，注销 SRV，释放 ready 纹理和 fallback。 依赖 backend 传入显式 Gfx ctx。

内部结构

texture_states

使用 SlotMap<AssetTextureHandle, LoadStatus> 保存纹理状态。

textures

使用 SecondaryMap 保存已 ready 的 AssetTexture。

texture_cache

路径到 handle 的缓存，用于避免重复加载同一文件。

fallback_texture

1x1 粉色纹理，覆盖 loading、uploading、failed 的访问路径。

AssetLoader

dispatch thread + rayon pool，负责磁盘 IO 与图片解码。

AssetUploadManager

管理 staging buffer、command buffer、transfer queue 与 timeline semaphore。

运行流程

Unloaded → Loading → Uploading → Ready 或 Failed
阶段 发生位置 行为
请求加载 AssimpSceneLoader::load_scene 材质包含 diffuse map 时调用 asset_hub.load_texture。
后台解码 AssetLoader image::open 后统一转为 RGBA8，返回 RawAssetData。
上传提交 AssetHub::update 收到 LoadResult::Success 后把状态改为 Uploading，提交给 upload manager。
上传完成 AssetUploadManager::update 查询 timeline semaphore，释放 staging 和 command buffer，返回完成的 image。
渲染准备 SceneManager::prepare_render_data 通过路径查询 texture，再从 bindless manager 解析 shader SRV handle。

数据视角

输入是材质中的路径字符串或 PathBuf。后台线程输出 RawAssetData { pixels, extent, format, handle, mip_levels }。 上传完成后，AssetHub 把 GfxImage 注册为 image handle，再创建 image view 并注册 SRV。

当前图片格式固定为 R8G8B8A8_UNORM，mip 固定为 1；AssetHub 内部也有 is_srgb 与 mip_levels 的 TODO，说明格式元数据还未完整从加载数据传递。

依赖关系

方向 对象 说明
直接依赖 truvis-gfx 创建 image、staging buffer、command buffer、timeline semaphore。
直接依赖 truvis-render-interface 使用 GfxResourceManager、BindlessManager 与资源 handles。

反向使用 truvis-render-runtime 创建、每帧更新、销毁 AssetHub。
反向使用 truvis-world prepare 阶段从 AssetHub 查材质贴图。

生命周期与状态

RenderRuntime::new 创建 AssetHub。RenderRuntime::begin_frame 调用 asset_hub.update，把资产更新内聚到帧开始。关闭时 RenderRuntime::destroy 在 GPU idle 后调用 asset_hub.destroy，保证 upload manager、ready textures 和 fallback 都在 Gfx 销毁前释放。

生命周期风格符合项目约束：长期资源不保存 typed Gfx ctx，释放时由 owner 显式传入上下文。

并发与性能

已异步化

文件读取和 CPU 解码运行在 dispatch thread 与 rayon worker pool，不阻塞加载调用点。

仍在 render thread

上传 command buffer 录制、submit、view 创建和 bindless 注册发生在 begin_frame。

上传同步

使用 timeline semaphore 查询完成进度，pending 队列有序回收 staging buffer 和 command buffer。

潜在瓶颈

当前每个纹理一次 command buffer 和一次 submit，批量纹理完成时可能造成单帧尖峰。

风险与建议

最大边界风险：AssetHub 直接调用 BindlessManager::register_srv/unregister_srv， 使资产层知道 shader-visible descriptor 策略。项目设计讨论已明确认为这个边界过宽。
问题 影响 建议
AssetHub 绑定 bindless 策略 资产层承担 renderer/backend 责任，后续引入 texture binding cache 会受阻。 让 AssetHub::update 返回 AssetTextureEvent，bindless 注册移到 backend。
SceneManager 同时依赖 asset 和 bindless CPU scene 知道 GPU view 和 shader binding 细节。 材质加载阶段保存 AssetTextureHandle，prepare 阶段通过 resolver 解析。
get_texture_by_path 使用 unwrap 调用顺序错误会直接 panic。 长期改成返回 Option 或在模型加载阶段消除路径查询。
上传粒度较细 大量纹理同帧完成 IO 时，render thread 可能出现提交尖峰。 在 AssetUploadManager 内部批量合并同帧 upload。
