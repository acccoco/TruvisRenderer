## 材质管理：全量上传 vs 增量 dirty 方案

### 当前流程

```
每帧 tick:
SceneManager::prepare_render_data()  →  RenderData（全量快照）
                                            ↓
GpuScene::upload_render_data()       →  全量写入 stage buffer → copy → device buffer
```

- `SceneManager.all_mats`（`SlotMap<MaterialHandle, Material>`）持有 CPU 侧材质数据
- 每帧调用 `prepare_render_data()` 遍历所有材质，构建 `RenderData` 临时快照
- `GpuScene::upload_material_buffer()` 将快照全量写入 stage buffer，再 `vkCmdCopyBuffer` 到 device buffer
- GPU buffer 中的 slot 由遍历顺序隐式决定，**不稳定**——增删材质后同一材质的 index 可能变化


### 目标方案：增量 dirty 上传

核心思路：
- 材质注册时分配稳定的 GPU buffer slot，直到删除才释放
- 每帧只检查 dirty 标记，仅上传发生变化的材质
- 支持等待材质依赖的 texture 异步加载完成

这是主流引擎的常见做法（如 Unreal 的 `FMaterialRenderProxy`），项目中的 `BindlessManager` 已验证了相同模式。


### 对比

| 维度 | 全量上传（当前） | 增量 dirty（目标） |
|------|-----------------|-------------------|
| **CPU 开销** | $O(n)$ 每帧，n = 材质总数 | $O(k)$ 每帧，k = dirty 数量，稳态下 $k \approx 0$ |
| **传输带宽** | 每帧全量 copy | 仅 dirty slot 需要更新 |
| **slot 稳定性** | 不稳定（遍历顺序决定） | 稳定（注册时分配，删除才释放） |
| **texture 依赖** | 无等待机制，`prepare_render_data` 时必须就绪 | 可延迟：texture 未就绪时 slot 保持 placeholder |
| **实现复杂度** | 简单 | 中等（dirty list、free list、FIF 回收） |


### 架构方案选型

#### 方案 1：维护在 SceneManager + GpuScene

```
SceneManager  ─── CPU 材质数据 + dirty 标记
GpuScene      ─── GPU buffer + slot 映射
```

- 职责分散：dirty 状态在 SceneManager，slot 分配在 GpuScene，两者必须协调
- 接口耦合：修改材质时需同时通知两个模块
- 与 `BindlessManager` 的设计风格不一致

#### 方案 2：独立的 MaterialManager（推荐）

```
MaterialManager  ─── CPU 材质参数 + GPU slot 映射 + dirty list + 增量上传
```

- **内聚**：材质的 CPU 数据、GPU slot、dirty 状态、上传逻辑全部聚合
- **一致**：与 `BindlessManager` 设计模式相同，已在项目中验证
- **可扩展**：支持 texture 异步依赖内化，未就绪时使用 placeholder handle，就绪后标记 dirty
- **简化 GpuScene**：不再需要 `material_buffer` / `material_stage_buffer`，从 MaterialManager 获取 device address 即可

#### 结论

选择**方案 2**。


### MaterialManager 设计草图

与 `BindlessManager` 的差异：
- `BindlessManager` 写入 **descriptor set**（`vkUpdateDescriptorSets`，CPU 操作）
- `MaterialManager` 写入 **SSBO buffer**（`vkCmdUpdateBuffer` 或局部 stage copy，需要 `cmd`）
- 单个 `PBRMaterial` 约 64 bytes，`vkCmdUpdateBuffer` 适用（上限 65536 bytes）


### 对现有模块的影响

- **SceneManager**：不再持有 `all_mats`，改为仅存储 `MaterialHandle` 引用
- **GpuScene**：移除 `material_buffer` / `material_stage_buffer`，从 `MaterialManager` 获取 device address 填入 `GPUScene` struct
- **RenderData**：材质部分仅需要 slot index（已稳定），不再需要全量拷贝 `MaterialRenderData`

同样的模式可推广到 `MeshManager`、`LightManager`，逐步将 `GpuScene` 简化为纯粹的 **场景级 GPU 数据聚合器**。
